## Why this milestone blocks the app layer

The codebase has accumulated meaningful duplication:
- Every app that needs MOA-shaped state gets MOA bundled
  unconditionally into its binary (via `MOA_AP_SOURCE` in
  codegen). Apps that don't need MOA still pay for it.
- Cross-app helper patterns (tagged-accumulator parsing, directory
  walks, JSON shapes) live in the std seed because there's no
  other shared home. The std seed has grown to absorb friction
  that should live in libraries.
- Apps that want to share code with sibling apps either copy
  source or route through the std seed (inappropriate for
  domain-specific shared code).

The fix: source-level imports of vendored library directories.
After this milestone, MOA becomes the first vendored library
(`lib/moa/`); apps that want it `import "lib/moa" as moa;` and
get `moa::*` in scope. Apps that don't, get a smaller binary.

## Locked-in design decisions (don't relitigate)

- **Source-level, vendor-the-source model.** A library is a
  directory of `.ap` files (per-dir seed per F.19). No package
  manager, no registry, no fetch, no versioning, no lockfile.
  Libraries are vendored into a project's `lib/` directory (or
  the workspace's `lib/`).
- **Syntax**: `import "lib/foo" as foo;` — alias is required.
  Bare `import "lib/foo";` (no alias) is a parse error. Same
  forcing-function discipline as the v1.x-3 no-default-sub-mode
  rule and the v1.x-FORM-2 two-channel rule: the user names the
  namespace at the import site.
- **Reference syntax**: `foo::Bar` for the locus / type `Bar`
  exported from library `foo`. Mirrors `std::lang::Morpheme` and
  `moa::BraidId` shapes.
- **Library path resolution order** (first match wins):
  1. `<entry-dir>/<path>/` — relative to the importing file's
     directory
  2. `<workspace-root>/<path>/` — fallback for project-wide
     vendored libs
  No `$APERIO_PATH`, no global lib cache. Two-step resolution
  keeps things predictable.
- **Auto-mangle imported decls at parse time** with prefix
  `__lib_<alias>_<filestem>_<name>`. Same shape as the
  `__StdLangMorpheme` / `__MoaBraidId` mangling that std + moa
  use today, but generated automatically instead of hand-spelled.
- **No `pub` / `export` keyword.** Everything top-level in an
  imported seed is exported. Matches the current intra-seed
  visibility model. Add `pub` later if a workload demands it.
- **No re-exports.** If library A imports library B, B is NOT
  visible to A's importers. Strict barrier.
- **Cycles are parse errors.** Reuse the `visited` set already in
  `parse_with_imports`.

## What already exists (don't re-derive)

- **AST**: `pub struct Import { path: String, alias: Option<String>, span: Span }`
  at `crates/aperio-syntax/src/ast.rs:18`. `Program.imports`
  carries them.
- **Lexer**: `TokenKind::Import` at `lexer.rs:63`, keyword wired
  at line 400.
- **Parser**: `parse_import()` at `parser.rs:255` handles
  `import "path";` and `import "path" as alias;` both. The alias
  field is parsed but currently dead syntax — nothing reads it.
- **CLI**: `parse_with_imports()` at `aperio-cli/src/main.rs:144`
  walks imports recursively, follows `import "foo"` to
  `<dir>/foo.ap`, has cycle detection via `visited` set. Today
  this does a FLAT MERGE — imported decls just become part of
  the program's top-level. No namespace separation.
- **Codegen path-rename machinery**: `STDLIB_PATH_RENAMES` at
  `codegen.rs:475` and `MOA_PATH_RENAMES` at `codegen.rs:514`
  map `["std", "lang", "Morpheme"]` → `"__StdLangMorpheme"`.
  `stdlib_mangled_for_path()` dispatches by first segment. This
  is the SHAPE imports extend — auto-populated table per
  imported library.
- **Per-dir seed model (F.19)**: `aperio build apps/foo` already
  treats every `.ap` in the directory as one bundle. Imports
  reuse this for the library side.

## What's missing (the actual work)

1. **Parser tightening**: `import "path";` (no alias) becomes a
   parse error per the forcing-function rule. Aliases were
   already optional in the AST; change the parser to require
   them at the surface and add a tracking allow-list for the
   one historical bare form (`import "std";` if any examples
   exist — likely none).
2. **Directory imports**: today `import "foo"` resolves to
   `<dir>/foo.ap`. Extend to also try `<dir>/foo/` as a directory
   bundle (read every `.ap` in that directory as one seed).
   Single-file imports stay supported.
3. **Library path resolution**: when relative-to-entry doesn't
   find the import, fall back to `<workspace-root>/<path>/`.
   Workspace root is the directory containing `Cargo.toml` at
   the top, or whichever heuristic is least fragile (probably
   "search upward for `Cargo.toml`" since the workspace is the
   Rust monorepo).
4. **Auto-mangler**: build the `crates/aperio-codegen/src/mangle.rs`
   walker that yesterday's session sketched and backed out
   (~150 LOC, AST-level rewrite). Per-file mangling: collect
   top-level decl names from each imported `.ap` file, rewrite
   decl-sites + use-sites with prefix `__lib_<alias>_<file>_`.
   Same walker extends to std + moa eventually (separate cleanup
   milestone; not in scope here).
5. **Path-rename table**: register `["<alias>", "<Name>"]` → mangled
   form for every exported top-level decl from each imported
   library. Auto-populated at parse-bundle time. The existing
   `stdlib_mangled_for_path()` dispatcher gains a third arm.
6. **MOA migration**: pull MOA out of `MOA_AP_SOURCE` /
   `MOA_PATH_RENAMES` unconditional bundling. Make MOA's source
   live at `lib/moa/` (it already does) and require apps to
   write `import "lib/moa" as moa;` to use it. Existing apps that
   use MOA (`apps/market-book/`, etc.) get an `import` line
   added.

  source files exercise (TypeExpr, Expr, Stmt, Pattern, all

  do NOT auto-derive again; it's substantial work to get right.
- Unit tests on synthetic Program inputs (no codegen integration
  yet).

**PR3: CLI + codegen plumbing for cross-seed imports**
- Extend `parse_with_imports` to:
  - Try `<dir>/<path>.ap` (existing single-file)
  - Try `<dir>/<path>/` (new directory bundle)
  - Try `<workspace-root>/<path>/` (new workspace fallback)
- Tag each parsed program with its (alias, seed-root) for
  downstream mangling.
- Codegen merges imported programs after applying the mangler;
  auto-populates the path-rename table from each imported seed's
  exported names.
- End-to-end test: a tiny library `crates/aperio-codegen/tests/fixtures/lib-toy/`
  with one or two locuses; a fixture app that imports it; verify
  build + run produces expected output.

**PR4: MOA migration**
- Move `MOA_AP_SOURCE` / `MOA_PATH_RENAMES` infrastructure into
  the generalized auto-mangle path (still bundled by default
  for one transitional step).
- Apps that use MOA add `import "lib/moa" as moa;`.
- Once all consumers have the import, stop the unconditional
  bundling. Smaller binaries for apps that don't need MOA.

**PR5: Spec + docs**

## Open design questions (settle before PR3)

1. **Workspace root detection.** Search upward for `Cargo.toml`?
   For a `.aperio-workspace` sentinel? For now use Cargo.toml since
   the workspace IS the Rust monorepo, but note the
   limitation — apps shipped standalone won't have one.
2. **What `import "moa" as moa;` (no `lib/` prefix) does.** If
   `moa` resolves to a top-level `lib/moa/` after the migration,
   the `lib/` prefix is redundant. Decide whether to require it
   or treat `lib/` as implicit search-prefix.
3. **Error reporting when an imported library has a parse error.**
   Today errors get reported with the importer's path. Need to
   attribute parse errors to the imported file's path.
4. **Test fixture location.** New `crates/aperio-codegen/tests/fixtures/`
   directory? Or scattered in existing test files?

## Discipline notes

- Don't bundle unrelated changes. Use `git add <specific files>`.
- v1 ships source-level imports only. **DON'T be tempted to add
  versioning, fetch, or a package manager** — those are
  multi-week work and not in scope. The vendor-the-source model
  is the v1 commitment; defer the rest until a real friction log
  demands it.
- **DON'T add `pub`/`export` keywords.** Everything top-level is
  exported. Adding fine-grained visibility doubles the design
  surface and isn't on the v1 roadmap.
- **DON'T migrate std seed away from hand-mangling in this
  milestone.** The auto-mangler is shared infrastructure; using
  it for imports is in scope. Migrating std + moa to also be
  auto-mangled is a separate cleanup that should land after the
  import machinery is proven on user libraries.
- After v1.x-IMPORT closes, the next planned items are:
  - Pull MOA out of always-bundled (PR4 above)
  - Phase 4: apps/+examples/ rollup using imports for shared
    helper libraries
  - v1.x-9 (closures with capture) — still design-gated on MS2
  - v1.x-FORM-4 (`@form(hashmap)` / `@form(ring_buffer)`) — still
    design-gated on driver workload

## Why source-level vendoring is the right v1 shape

A real package manager (registry, fetch, semver, lockfile) is
several months of work. The friction this milestone unblocks is
"can libraries exist at all" — not "can we deduplicate
dependencies across projects on the same machine." Vendoring is
how C, early Go, and many other languages bootstrapped their
library ecosystems before package managers existed. Aperio's
file-based, dir-seed model is already well-suited to it.

When a real workload surfaces friction that vendoring causes
(version skew across projects, duplicate sources on disk, manual
update toil), that's the signal to design the package manager.
Not before.
