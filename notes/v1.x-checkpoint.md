- `[ ]` **3. Recognition projection class proper backing** —
  design RESOLVED 2026-05-11; implementation pending

  **Design decision (Design-aligned, user-confirmed):**
  Recognition's storage discipline is an axis with FOUR named
  sub-modes — same pattern as projection class itself,
  schedule class, and F.22 slot kinds. The user *writes down*
  which commitment they're making per-locus.

  Syntax:
  ```
  : projection recognition(cap=N, <sub-mode>)
  ```

  Sub-modes:
  - `fixed_cell(bytes=N)` — each child fits in N bytes; overflow
    is a hard runtime error. Bounded state you can size up front.
  - `spillover(bytes=N)` — fixed N-byte cell + malloc fallback
    with one-time warning. Graceful degradation under load.
  - `summary_only` — children carry zero per-instance state; all
    allocations live in parent's arena.
  - `shared_slab(bytes=N)` — one bump arena of N bytes serves
    all children; per-child dissolve is no-op; whole slab frees
    at parent dissolve.

  Default sub-mode if omitted: TBD (proposal: `fixed_cell(bytes=512)`).
  Decision deferred — user hasn't picked yet.

  **Ship cadence (v1 / v1.x):**
  - v1: parse all four sub-modes; codegen ships `fixed_cell` and
    `shared_slab` (cleanest implementations); reject `spillover`
    and `summary_only` with "v1.x pending" diagnostic (mirrors
    v1.x-4 surface pattern for `as_parent_for`).
  - v1.x: ship `spillover` (needs malloc-fallback + warning
    machinery).
  - v1.x or later: ship `summary_only` (needs type-system rule
    "no child arena allocation").

  **C runtime work:**
  - `lotus_recpool_fixed_*` — bitmap-backed fixed-cap pool of
    fixed-size cells; acquire/release O(1) via bitmap scan;
    no malloc in steady state.
  - `lotus_recpool_slab_*` — single bump arena with bitmap of
    "child boundaries" so individual child dissolve is no-op
    but parent dissolve frees the whole slab.

  **Codegen work:**
  - Extend ProjectionClass to carry sub-mode + params.
  - Route child-locus instantiation through the matching
    recpool primitive when parent is recognition-class.
  - Accept the existing `lotus_arena_t*` ABI on the child side
    by wrapping the recpool cell in a fixed-cap arena variant
    (children's body code stays projection-class-agnostic per
    the F.22 architectural invariant).

## Form-extending (Design mechanics at new levels)

- **REMOVED 2026-05-12 — 6. Result + `?` operator**
  - Cut from the roadmap after a design pass. `?` is pure
    desugaring (`r.is_err { return r.err; }`); the load-bearing
    part is `Result<T, E>` as a discipline that forces callers
    to acknowledge errors at the type level.
  - Aperio already has failure-propagation-upward at the
    **locus** level via `bubble` / `on_failure` — the Design's
    failure-propagation-upward mechanic expressed structurally.
    Adding value-level Result would create a second, parallel
    mechanism for the same thing (parametric option for what
    is structurally already covered — exactly the shape The
    Design counsels against).
  - The current Aperio pattern is clean:
    - `parse_int(s)` returns `0` on failure (sentinel-with-default).
    - `can_parse_int(s)` is the explicit discriminator.
    Caller picks the appropriate pair — no type-system tax on
    the common case where 0-on-failure is fine.
  - No friction-log entry asks for value-level Result. If a
    future workload demands one, revisit the design call —
    but don't ship speculatively.

- `[#]` **7. Match exhaustiveness via case folding** —
  no concrete friction driver yet
  - check.rs `match_is_exhaustive` currently exhaustive on
    Bool (both literals) and Enum (all variants), permissive
    on Unknown / unrecognized names, requires catchall
    elsewhere. That's sound today.
  - The v1.x-7 framing was forward-looking — fold `0|1|2|3` +
    `4..=10` into `0..=10` and check coverage; let `0..=i64::MAX`
    skip catchall on Int. No friction-log entry calls for it.
  - Defer until a workload writes ranges that the current
    "require catchall" rule rejects unhelpfully.

- `[x]` **8. `type` records hold `fn(...)` fields** — commit TBD
  - Three fixes (not one):
    1. Parser: `parse_struct_field` + `parse_struct_init` use
       `expect_member_name` instead of `expect_ident`; extended
       `try_member_keyword_as_name` to admit capacity / birth /
       accept / run / drain / dissolve as field names.
    2. Codegen: `lower_external_method_call` gained a TypeRef +
       FnPtr-field branch ahead of the LocusRef path; GEPs the
       field, loads the fn pointer, emits indirect call via
       `emit_fnptr_indirect_call`.
    3. No typecheck change needed — once parser admitted the field
       and codegen had the dispatch path, existing `fn(...)` type
       representation was sufficient.
  - 3 codegen tests in `type_record_fn_pointer_field.rs`
    (zero-arg, int-arg with mixed struct payload, dispatcher
    pattern selecting among records).
  - Friction marked LANDED 2026-05-11.

- `[x]` **10. String interpolation** — commit TBD
  - Opt-in `f"..."` prefix (NOT plain `"..."`) so existing
    `.ap` sources with literal `{...}` content (JSON, etc.)
    keep working unchanged. Backward-compat preserved.
  - Lexer recognizes `f"..."`, splits body into FStringPart::Lit
    / Interp segments; `{{` / `}}` are literal braces.
  - Parser sub-parses each Interp body as a full Aperio
    expression and wraps in `to_string(...)`; whole f-string
    lowers to a left-folded `+` chain.
  - Works in both interpreter and codegen (no new lowering;
    just AST desugaring at parse time → existing string + and
    to_string paths).
  - 8 tests in `fstring_interpolation.rs` (ident, int, multi,
    dotted member, arithmetic expression, escaped braces,
    bare-interpolation, plain-string-backcompat).
  - Spec: tokens.md updated with f-string entry.
