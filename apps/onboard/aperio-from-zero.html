Aperio from Zero

A lotus grown one layer at a time.

sixteen chapters · ends with a real multi-binary app

What this book is

A guided walk from the smallest valid Aperio program to a running, two-binary chat-fanout system. Sixteen chapters, each introducing one or two language features and accumulating toward the capstone. By the end you will have built — from scratch — a terminal chat client and a TCP gateway that fan-out messages, both compiled from the same shared seed.

Each chapter ends with code you can run. Chapters carry a status: ships means it compiles against today's toolchain; partial means parsed and accepted but with a runtime component still landing; designed means the surface is specified but the implementation arrives later.

Pair this with why-aperio for the rationale. Pair it with spec/ for the formal language reference. Pair it with the existing apps/ directory for worked examples beyond the book's scope.

One convention threads through every example: the outermost encapsulation of an app is a locus, never a bare fn main() with helpers. main reads argv (when needed) and instantiates the outer locus; the locus's lifecycle owns the run. This is the apps-are-loci rule from notes/agent-onboarding/aperio-styleguide.md: when everything is a lotus, everything is reusable, and the shape is consistent at every layer including the top.

· · ·Part I · A First Locus

Chapter 1 ships

Hello, locus

The smallest valid Aperio program is one locus and one main.

Aperio's fundamental unit is the locus — a thing with state, lifetime, and a place in a tree. Every Aperio program is built out of loci, from the outermost application down to the smallest internal coordinator. Even a free function lives inside an implicit locus. The locus is what your program is made of.

Here is the smallest program that exists in Aperio.

apps/ch01/main.ap

locus HelloL {
    params { greeting: String = "hello, world"; }
    birth() { println(self.greeting); }
}

fn main() {
    HelloL { };
}
$ aperio run apps/ch01/main.ap
hello, world

Three lines of content. A locus called HelloL (the trailing L is a convention we'll keep — it marks types that are loci, as distinct from plain shapes). The locus declares one param — its state — with a default value. It declares a birth() method that runs once, synchronously, at instantiation.

The body of main instantiates the locus with HelloL { };. This allocates the locus's region, runs its birth(), and — because HelloL has nothing to do after birth — dissolves it at the end of the statement. Region freed wholesale, exit 0.

Notice what is not here. No package, no import, no class. The locus is the unit of declaration. Everything else builds on it.

A locus is state plus lifecycle. The smallest program declares one, instantiates it from main, and runs to completion.

Next → Chapter 2 · Lifecycle

Chapter 2 ships

Lifecycle

Five named transitions: birth, accept, run, drain, dissolve.

A locus moves through five named lifecycle states in order:

birth() runs once at instantiation, synchronously.
accept(c: ChildL) runs each time a child is attached (Chapter 5).
run() is the steady-state loop, running until it returns naturally or is asked to drain.
drain() halts new work and lets in-flight finish.
dissolve() runs last, then the region is freed wholesale.

You declare any subset. Omitted methods get compiler-supplied defaults (e.g. default dissolve just frees the region). The compiler enforces ordering: you cannot accept before birth completes, cannot dissolve before drain.

apps/ch02/main.ap

locus TickerL {
    params { n: Int = 3; ms: Duration = 100ms; }

    birth()    { println("ticker birthing"); }
    run() {
        let mut i = 0;
        while i < self.n {
            println("tick " + i);
            std::time::sleep(self.ms);
            i = i + 1;
        }
    }
    drain()    { println("ticker draining"); }
    dissolve() { println("ticker dissolving"); }
}

fn main() {
    TickerL { n: 5, ms: 50ms };
}
$ aperio run apps/ch02/main.ap
ticker birthing
tick 0
tick 1
tick 2
tick 3
tick 4
ticker draining
ticker dissolving

The output is exactly the order the spec promises. birth, then the run loop, then drain, then dissolve. If you press Ctrl-C mid-run you'll see draining and dissolving fire then too — the runtime catches SIGINT and triggers drain() on the root, which cascades. We'll see that cascade properly in Chapter 7.

Notice main exits cleanly without any explicit wait() or join(). main's implicit locus is the parent of the anonymous TickerL, and a parent cannot return until its children have dissolved.

Lifecycle is the spine of every locus, including the implicit locus around main. Each method is a named transition; the compiler enforces the ordering.

Next → Chapter 3 · State & mutability

Chapter 3 ships

State & mutability

A locus's params are its state. Locals are immutable unless marked mut.

Most languages distinguish parameters (what you pass in) from state (what you carry between calls). Aperio collapses them: the params block declares both. Each entry is a birth-time default that may be overridden at instantiation and a runtime-mutable field accessible via self.x.

This matches Ruby's @foo instance-variable pattern: one bundle of named, typed state per object, no separate "state block."

apps/ch03/main.ap

locus CounterL {
    params { count: Int = 0; }

    birth() {
        let mut i = 0;
        while i < 5 {
            self.count = self.count + 1;
            i = i + 1;
        }
        println("final count: " + self.count);
    }
}

locus Ch3DemoL {
    run() {
        CounterL { };              // uses default count = 0
        CounterL { count: 100 }; // overridden at instantiation
    }
}

fn main() {
    Ch3DemoL { };
}
$ aperio run apps/ch03/main.ap
final count: 5
final count: 105

Two things at play. self.count is the locus's mutable state, settable directly. i is a local, declared with let mut because we need to reassign it in the loop. Drop the mut and the line i = i + 1 becomes a compile error.

Immutability is the default for bindings. This is Rust's model, not Go's. let x = 0; means x is fixed; let mut x = 0; means "I will reassign x." Mutability is a property of the binding, not the type.

Locus state lives in params and is accessed through self. Local bindings are immutable by default; mark mut where you mean it.

Next → Chapter 4 · Types & functions

Chapter 4 ships

Types & functions

Primitives, literals, user types, and the free fns that move between them.

Aperio has nine primitive types, all spelled in PascalCase: Int Uint Float Decimal String Bool Time Duration Bytes. PascalCase matters because the lowercase names (time, string, …) are free for stdlib namespaces like std::time::sleep. Type position picks up the capital; expression position the lowercase.

Literals carry types directly. 42 is Int. 3.14 is Float. 1.50d is Decimal (the d suffix matters — see §14 of the design rationale for why money is never Float). 5s, 100ms, 1h30m are Duration. `2026-05-11T12:00:00Z` (with backticks) is Time.

apps/ch04/main.ap

type Point { x: Float; y: Float; }

fn distance(a: Point, b: Point) -> Float {
    let dx = a.x - b.x;
    let dy = a.y - b.y;
    std::math::sqrt(dx * dx + dy * dy)
}

locus Ch4DemoL {
    run() {
        let p = Point { x: 0.0, y: 0.0 };
        let q = Point { x: 3.0, y: 4.0 };
        println("distance = " + distance(p, q));
    }
}

fn main() {
    Ch4DemoL { };
}
$ aperio run apps/ch04/main.ap
distance = 5

Three observations. First, type Point { ... } declares a plain record — no lifecycle, no birth. Types are shapes; loci are flow. Two declaration keywords, two clean categories. Second, the free fn distance takes ordinary value params and returns a typed value. Third, std::math::sqrt is a stdlib path — no import needed, the parser recognizes std::* directly.

Type inference: let dx = a.x - b.x; — the type of dx is inferred from the right-hand side (Float). Annotate explicitly with let dx: Float = … when you want the inference stated in source.

Types are shapes (type); loci are flow (locus). Free fns move values between shapes. PascalCase type names keep the stdlib namespaces clean.

· · ·Part II · Composition

Chapter 5 ships

Parent & child

A parent's accept(c) runs before its child's birth().

A locus instantiated inside another locus's lifecycle method becomes that locus's child. This is how the runtime tree grows. The parent declares accept(c: ChildType); that method runs before the child's region is allocated and birth runs. A parent that rejects (panics in accept) prevents the child from ever materializing.

apps/ch05/main.ap

locus GreeterL {
    params { name: String = "world"; }
    birth() { println("hello, " + self.name); }
}

locus CoordinatorL {
    accept(g: GreeterL) {
        println("coord: accepted greeter " + g.name);
    }
    run() {
        GreeterL { name: "alice" };
        GreeterL { name: "bob" };
    }
}

fn main() {
    CoordinatorL { };
}
$ aperio run apps/ch05/main.ap
coord: accepted greeter alice
hello, alice
coord: accepted greeter bob
hello, bob

Read the order carefully. accept runs first, on the parent, with the child's declared params visible. Then the child's region is allocated and its birth() runs. The parent gets a veto before any resources are committed.

The parent also gets self.children — an iterable of currently-attached children, typed [GreeterL] here. Add a closure or run-loop and you can walk it.

Children are loci instantiated inside a parent's lifecycle method. accept runs first, on the parent, and may reject. After accept, the child is registered in self.children automatically.

Next → Chapter 6 · The contract

Chapter 6 ships

The contract

Vertical flow is the only flow. The contract declares the typed surface.

A parent can see only what its child exposes; a child can see only what its parent declares it will consume. The compiler checks compatibility at compile time: if a child declares expose greeting: String and a parent declares consume greeting: String, the names and types must align. A mismatch fails to compile.

What you cannot do: have one sibling reach into another. The grammar admits no syntax for it; the type system has no rule that would type it. Lateral coordination flows through the parent — explicitly, via the bus or contract — or it does not happen.

apps/ch06/main.ap

locus GreeterL {
    params   { greeting: String = "hi"; }
    contract { expose greeting: String; }
}

locus CoordL {
    contract { consume greeting: String; }

    accept(g: GreeterL) {
        println("coord sees: " + g.greeting);
    }
    run() {
        GreeterL { greeting: "howdy" };
    }
}

fn main() {
    CoordL { };
}
$ aperio run apps/ch06/main.ap
coord sees: howdy

Try changing expose greeting: String to expose greeting: Int in the child and recompile. The parent's consume declaration still says String; the compiler refuses to link them. Contract violations are structural, and they are caught at compile time.

Defaulting: if a contract entry's name matches a params entry, the param is the implementation. You can add a member fn returning the contracted type to override or specialize per projection class. (See F.14; we'll meet projection classes at the end of the book.)

The contract is the typed surface between a locus and its parent. The compiler enforces compatibility. Siblings cannot reach each other — laterality goes through the parent.

Next → Chapter 7 · Drain cascade

Chapter 7 ships

Drain cascade

SIGINT triggers drain() on the root. The cascade falls out of the tree.

When you press Ctrl-C, the runtime catches SIGINT and calls drain() on the root locus. drain is always cascading: it recursively drains every child, depth-first, waits for them to finish, then drains itself.

This is the F.4 commitment. No separate cascade syntax. No context plumbing. No errgroup. The shape of the tree is the shape of the shutdown.

apps/ch07/main.ap

locus LeafL {
    params { id: Int; }
    birth()    { println("  leaf " + self.id + " birth"); }
    drain()    { println("  leaf " + self.id + " drain"); }
    dissolve() { println("  leaf " + self.id + " dissolve"); }
}

locus TreeL {
    accept(c: LeafL) { }
    birth()    { println("tree birth"); }
    run() {
        LeafL { id: 1 };
        LeafL { id: 2 };
        LeafL { id: 3 };
        std::time::sleep(10s); // wait for Ctrl-C
    }
    drain()    { println("tree drain"); }
    dissolve() { println("tree dissolve"); }
}

fn main() {
    TreeL { };
}
$ aperio run apps/ch07/main.ap
tree birth
  leaf 1 birth
  leaf 2 birth
  leaf 3 birth
^C
  leaf 3 drain
  leaf 3 dissolve
  leaf 2 drain
  leaf 2 dissolve
  leaf 1 drain
  leaf 1 dissolve
tree drain
tree dissolve

Shutdown order is reverse-of-instantiation, leaves first. Leaf 3 drains and dissolves before leaf 2; both finish before leaf 1; all three before the tree drains itself. The order is deterministic and falls out of the shape — there is no code here that says "shut down in this order." The shape said it.

drain() is always cascading. SIGINT triggers it on the runtime root and the cascade walks the tree depth-first, leaves-first. You don't write the cascade; you write the tree.

Next → Chapter 8 · Recovery

Chapter 8 partial

Recovery

When a child fails, its parent decides what happens next.

Failure flows up. A child that panics, or whose run() throws, invokes its parent's on_failure(c, err) handler. The parent then chooses among recovery primitives — restart, restart_in_place, quarantine, bubble, dissolve — by calling the one it wants. This is OTP's "let it crash + supervisor" shape, surfaced as language.

apps/ch08/main.ap

locus FlakyL {
    params { id: Int; }
    birth() {
        if self.id == 2 {
            panic("flaky " + self.id + " failed");
        }
        println("flaky " + self.id + " ok");
    }
}

locus SuperL {
    accept(c: FlakyL) { }
    on_failure(c: FlakyL, err: Error) {
        println("super: child " + c.id + " failed; restart");
        restart_in_place(c);
    }
    run() {
        FlakyL { id: 1 };
        FlakyL { id: 2 }; // will panic
        FlakyL { id: 3 };
    }
}

fn main() {
    SuperL { };
}

Five primitives, increasing in scope: restart_in_place re-runs birth with the same arena preserved; restart re-instantiates the child fresh; quarantine pauses the child for inspection; bubble forwards the failure to this locus's parent; dissolve destroys the child.

Default on_failure is bubble(err). The runtime root's default on_failure is process exit with a structured violation report.

Recovery primitives parse and dispatch in current Aperio; the runtime for restart_in_place and quarantine in the full m-series scheduler is still being finished. For most cases today, bubble and dissolve are load-bearing; the others are accepted but their full mid-flight semantics are landing.
Failure flows up. on_failure(c, err) is the parent's policy decision. Five primitives cover the range from in-place restart to outright dissolution.

Chapter 9 ships

Interfaces

Same surface, three loci. Structural satisfaction — no impl I for L.

An interface in Aperio is a Go-shaped structural interface: a named set of method signatures. Any locus whose method set is a superset structurally satisfies the interface. There is no impl Sink for StdoutSink declaration. Satisfaction is implicit; the compiler checks it at every call site.

apps/ch09/main.ap

interface Sink {
    fn write(s: String);
    fn line(s: String);
}

locus StdoutSinkL {
    params { }
    fn write(s: String) { print(s); }
    fn line(s: String)  { println(s); }
}

locus StringSinkL {
    params { buf: String = ""; }
    fn write(s: String) { self.buf = self.buf + s; }
    fn line(s: String)  { self.buf = self.buf + s + "\n"; }
}

fn render(sink: Sink) {
    sink.line("hello");
    sink.line("world");
}

locus Ch9DemoL {
    run() {
        let a = StdoutSinkL { };
        let b = StringSinkL { };
        render(a);
        render(b);
        println("captured: " + b.buf);
    }
}

fn main() {
    Ch9DemoL { };
}
$ aperio run apps/ch09/main.ap
hello
world
captured: hello
world

One render function works for both sinks. The compiler resolves the interface coercion at each call site, synthesizes a per-(locus, interface) vtable, and dispatches through it. No tagged loci. No branching inside StringSinkL for "am I a stdout or a string."

Structural interfaces let one fn body work for any locus whose method set covers the interface. The compiler does the rest.

Next → Chapter 10 · The bus

Chapter 10 ships

The bus

Typed pub-sub between loci. Subscribe declaratively; publish with <-.

The contract is for the parent-child surface. The bus is for everything else — cross-locus, cross-process, eventually cross-machine. Each locus declares its bus interface in source: which subjects it subscribes to and what type of payload arrives, which subjects it publishes and what type it sends. The compiler typechecks each <- at its call site.

apps/ch10/main.ap

type Event { kind: String; payload: String; }

locus ListenerL {
    bus {
        subscribe "app.events" as on_event of type Event;
    }
    fn on_event(e: Event) {
        println("got " + e.kind + ": " + e.payload);
    }
}

locus EmitterL {
    bus {
        publish "app.events" of type Event;
    }
    run() {
        "app.events" <- Event { kind: "login", payload: "alice" };
        "app.events" <- Event { kind: "click", payload: "home" };
    }
}

locus BusDemoL {
    run() {
        ListenerL { };
        EmitterL { };
    }
}

fn main() {
    BusDemoL { };
}
$ aperio run apps/ch10/main.ap
got login: alice
got click: home

<- is a statement, not an expression — it produces no value. The left side names a subject declared in this locus's publish set; the right side is the typed payload. Reception is structural — the subscribe SUBJECT as HANDLER declaration binds an incoming message to a member fn on the locus.

Wildcards: subscribe "log.app.**" catches a whole sub-tree of subjects. ** matches zero or more dot-separated segments. Useful for telemetry routing and the std::log hierarchy.

The bus is the framework's lateral interface, kept typed by the language. Subscribe is structural; publish is the <- operator. Subjects are namespaced; the runtime routes them via per-deployment transports.

Next → Chapter 11 · Networking

Chapter 11 ships

Networking

A TCP server is one stdlib locus plus a callback.

std::io::tcp::Listener is a stdlib locus. You instantiate it with a port and a callback; the framework gives you a real listening socket, accept loop, and a Stream per connection. Drain cascade on SIGINT closes every stream automatically.

apps/ch11/main.ap

fn on_connection(s: std::io::tcp::Stream) {
    s.send("welcome to the echo server\n");
    loop {
        let line = s.recv();
        if line == "" { break; }
        s.send("echo: " + line);
    }
}

locus EchoServerL {
    params { port: Int = 9000; }
    run() {
        std::io::tcp::Listener {
            port: self.port,
            on_connection: on_connection
        };
    }
}

fn main() {
    EchoServerL { };
}
$ aperio run apps/ch11/main.ap &
$ nc localhost 9000
welcome to the echo server
hello
echo: hello
^C

Notice the shape: on_connection is a free fn, passed by name to the Listener's on_connection param. This works because Aperio has first-class function pointers (m80). Each connection produces its own Stream locus, whose dissolve() closes the fd; SIGINT triggers the cascade and every connection cleans up.

Compare to the SIGINT-cascade discussion in why-aperio's §V. The shape is the same; here you see it actually compose with stdlib.

std::io::tcp::Listener hands you a connection callback. The per-connection Stream locus manages its own fd; the drain cascade closes everything on shutdown.

Next → Chapter 12 · Perspectives

Chapter 12 partial

Perspectives

A schema both binaries compile from. The bus carries values; the source carries the type.

A perspective is a serializable parameter bundle within a compiled-in shared schema. Producer and consumer compile from the same Aperio source, so the perspective is the wire contract. The bus carries parameter values; no version negotiation; no Protobuf .proto file separate from the code; no "we'll cut over Thursday" deployment dance.

apps/ch12/chat.ap (shared)

perspective ChatMessage {
    params {
        room: String;
        sender: String;
        body: String;
        ts: Time;
    }
    stable_when {
        return self.body != "" && self.sender != "";
    }
    serialize_as ChatMessageV1;
}

stable_when is a predicate the runtime evaluates before shipping or accepting a perspective. Useful for "this thing isn't ready yet" gates — the rotation handshake never sees a half-baked message.

The cross-binary trick: two Aperio binaries both compile from a directory that contains chat.ap (the perspective declaration) plus each binary's main.ap. F.19 (per-directory seed) makes this a one-flag affair. The perspective compiles into both as the same type, with the same field layout, the same stable_when, the same serialization. There is nothing for the wire format to disagree about.

Perspectives parse, typecheck, and the stable_when predicate runs in the current toolchain. Cross-binary hot-load over the bus (the full schema-aware adapter that handles versioning gracefully) is the next milestone after Phase 6 substrate work. Single-binary perspectives ship and serialize today.
A perspective is a serializable type that both producer and consumer compile from the same source. No protocol handshake — the source is the schema.

Chapter 13 ships

A chat room

A room is a locus that accepts participants. Membership falls out of self.children.

Start with the room. A RoomL accepts ParticipantL children, each carrying a name. Membership is a property of the runtime tree — when a participant joins, the room's accept runs; when one leaves, dissolve runs. The room's self.children is the live roster, typed and framework-maintained.

apps/chat/shared/room.ap

locus ParticipantL {
    params   { name: String; }
    contract { expose name: String; }
    birth()    { println(self.name + " joined"); }
    dissolve() { println(self.name + " left"); }
}

locus RoomL : projection chunked {
    params   { id: String; }
    contract { consume name: String; }

    accept(p: ParticipantL) {
        println("[" + self.id + "] welcome " + p.name);
    }
    run() {
        ParticipantL { name: "alice" };
        ParticipantL { name: "bob" };
    }
}

: projection chunked on the room signals moderate N — typical chat rooms have tens to low hundreds of concurrent members. Per-participant sub-regions with free-list bookkeeping. Change to rich for a two-person DM; change to recognition for a large channel with thousands of mostly-idle members. The body never changes.

Note where the participants are instantiated: inside RoomL.run(). That is what makes them children of RoomL — the parent-child relationship comes from the syntactic enclosure of the instantiation site, not from any room.add(...) call. The room's accept() fires for each, in order, before each child's birth() runs.

apps/chat/shared/main.ap

fn main() {
    RoomL { id: "#general" };
}
$ aperio run apps/chat/shared/
[#general] welcome alice
alice joined
[#general] welcome bob
bob joined
alice left
bob left

The room is the app's outer locus. Hardcoded membership in the run() body is a learning fixture — in Chapter 15 the gateway will spawn participants externally via per-TCP callbacks, and the room concept becomes mostly a subject namespace. The locus structure here teaches the room's shape; the network plumbing comes later.

Next → Chapter 14 · The terminal

Chapter 14 ships

The terminal

Binary one: read stdin, publish to chat.out; subscribe to chat.in, print.

The terminal client is the first of two binaries. It reads a line from stdin, publishes it as a ChatMessage on subject chat.out.<room>, and subscribes to chat.in.<room> to print messages it receives. The subscribe handler is a member fn on the terminal locus.

apps/chat/terminal/main.ap

locus TerminalL {
    params {
        room:   String = "#general";
        sender: String = "anon";
    }
    bus {
        subscribe "chat.in.#general" as on_in of type ChatMessage;
        publish   "chat.out.#general"             of type ChatMessage;
    }

    fn on_in(m: ChatMessage) {
        println("<" + m.sender + "> " + m.body);
    }

    run() {
        println("joined as " + self.sender + " in " + self.room);
        loop {
            let line = std::io::stdin::read_line();
            if line == "" { break; }
            "chat.out.#general" <- ChatMessage {
                room:   self.room,
                sender: self.sender,
                body:   line,
                ts:     std::time::now()
            };
        }
    }
}

fn main() {
    TerminalL {
        sender: std::env::var("USER")
    };
}

Three structural notes. First: the ChatMessage type used in the publish and subscribe declarations is the same perspective from Chapter 12 — shared via apps/chat/shared/. Second: the on_in handler is a member fn whose name matches the as clause. The runtime delivers incoming ChatMessage values directly into it. Third: std::env::var("USER") is the environment accessor (m77); it returns the var value or empty string.

Notice what we did not have to write. ChatMessage — the type we publish on the bus — is the same perspective declared in Chapter 12, pulled in simply by being in the same seed directory (F.19). When the gateway lands in Chapter 15, it will reuse ParticipantL from Chapter 13 unchanged, because a participant is a participant regardless of which side of the wire it's on.

This is what changes when every part of the system is a locus with a typed contract: every part becomes composable into a new context without modification. The contract is the surface, the surface is typed, and types compose by type-system rules that have nothing to do with what the locus does internally. Each new feature pulls existing loci into new arrangements rather than reinventing them.

Most codebases experience entropy increase — each new feature is harder than the last because the existing shape has to be retrofitted to accommodate. Aperio is the opposite. Momentum harmonizes: when everything is a lotus, everything is reusable. The room from Chapter 13, the perspective from Chapter 12, the bus subjects from Chapter 10 — they compose into the terminal, will compose again into the gateway, will compose again into whatever comes next.

The gateway

Binary two: terminate TCP, fan messages between connected terminals.

The gateway is the second binary. It listens on TCP, holds a RoomL per channel, and bridges between the per-connection stream and the bus. When a terminal sends a message, the gateway publishes it on the in-process bus to every other connected stream's subscriber. The room is local to the gateway; the bus is the substrate the terminals see.

apps/chat/gateway/main.ap

fn on_connection(s: std::io::tcp::Stream) {
    let sender = s.recv();   // first line is the user name
    let p = ParticipantL { name: sender };

    loop {
        let body = s.recv();
        if body == "" { break; }
        "chat.fanout.#general" <- ChatMessage {
            room: "#general", sender: sender,
            body: body, ts: std::time::now()
        };
    }
    // p dissolves at fn return (m82) — closes the participant
}

locus FanoutL {
    bus {
        subscribe "chat.fanout.#general" as on_fanout of type ChatMessage;
    }
    fn on_fanout(m: ChatMessage) {
        // re-publish on chat.in for every connected terminal
        "chat.in.#general" <- m;
    }
}

locus GatewayL {
    params { port: Int = 9001; }
    birth() { println("gateway up on :" + self.port); }
    run() {
        FanoutL { };
        std::io::tcp::Listener {
            port: self.port,
            on_connection: on_connection
        };
    }
    dissolve() { println("gateway down"); }
}

fn main() {
    let mut port = 9001;
    if std::env::args_count() > 1 {
        port = std::str::parse_int(std::env::arg(1));
    }
    GatewayL { port: port };
}

GatewayL is the gateway's outer locus — apps-are-loci, so the gateway has one. Its run() instantiates FanoutL (the bus bridge) and the TCP listener as children of GatewayL. SIGINT triggers drain() on the runtime root, which cascades into GatewayL, which cascades depth-first to its children: the Listener drains (closing every per-connection Stream — and each Stream's dissolve() closes its fd); FanoutL drains; GatewayL drains and dissolves. Every fd is closed, every arena freed, in order.

A note on the room model from Chapter 13: in this gateway the RoomL from ch13 doesn't appear directly. Each TCP connection's ParticipantL is a child of on_connection's implicit locus (per fn scope), not of any RoomL. The "room" in this distributed system is mostly a subject namespace — the chat.fanout / chat.in subjects identify the room. RoomL from ch13 is the in-process teaching model; the distributed shape is what you see here.

The flow: terminal stdin → gateway TCP → gateway publishes on chat.fanout → gateway's FanoutL republishes on chat.in → every terminal that subscribed gets it. Each terminal sees its own message echoed back along with everyone else's.

The gateway above uses the in-process bus to fan messages to every connected stream. To wire terminals across processes, a transport adapter (TCP, Unix socket, or a minimal broker) plugs the bus subject pattern into network delivery. The transport-binding layer is Phase 6 substrate work in progress; the source above is what the target shape looks like.

Audits & the garden

A closure invariant on message conservation. And where to go from here.

We have a chat-fanout system. There is one last move worth showing: an audit invariant in the source. The number of messages sent from terminals over some window must approximately equal the number of messages delivered to terminals — fanout multiplied by the number of connected terminals, within some tolerance. If the books don't balance, the gateway explodes structurally and the parent decides what to do.

apps/chat/gateway/main.ap (additions)

locus FanoutL {
    params {
        sent:      Int = 0;
        delivered: Int = 0;
    }
    bus {
        subscribe "chat.fanout.#general" as on_fanout of type ChatMessage;
    }

    fn on_fanout(m: ChatMessage) {
        self.sent = self.sent + 1;
        // republish + count deliveries — one per subscriber
        "chat.in.#general" <- m;
        self.delivered = self.delivered + self.parent.RoomL.children.length;
    }

    closure conservation {
        self.delivered ~~ self.sent * self.parent.RoomL.children.length within 2;
        epoch duration(10s);
    }
}

A closure declares a cyclic invariant the runtime checks at the declared epoch — every ten seconds here. The left side ~~ right side within tolerance is the band; failure flips the locus's exploded flag and the parent's on_failure receives a typed ClosureViolation at dissolve time. The books must balance.

Closure declarations parse and typecheck today; the runtime that evaluates them at the declared epoch and emits ClosureViolation is partially wired — the dissolve-epoch evaluation lands first, then tick-and-duration epochs. The shape above is the v1.0 target. For today, you can run the gateway and watch the counters with println — the data is there, the band check arrives shortly.
